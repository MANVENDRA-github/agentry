# Decisions

This document captures the design decisions behind agentry — what was chosen, what was rejected, and when each one is worth revisiting. Each decision is numbered so it can be referenced from commits, PRs, and other docs.

The goal is not to relitigate. It is to keep the rationale fresh enough that future contributors (and future-me) do not re-derive it from scratch six months from now. New decisions get appended with the next number. Existing entries are amended in place when superseded, with a short note about what changed and when.

Companion reading: [`architecture.md`](architecture.md) for how the system is structured, [`authoring.md`](authoring.md) for how to add new agents and skills.

## Architecture

### D1: Source-of-truth directories, generated harness directories

**Decision.** Author content lives in top-level directories — `agents/`, `skills/`, `commands/`, `rules/`, `hooks/`, `mcp/` (the latter three added in later releases). Harness-specific layouts (`.claude/`, `.cursor/`, `.codex/`, `.opencode/`, `.claude-plugin/`) are generated by `scripts/sync-harnesses.js` and treated as build output.

**Why.** The opposite shape — authoring directly into `.claude/` and `.cursor/` — guarantees that the two copies of a `code-reviewer` agent drift the first time someone edits one and forgets the other. A sync step makes drift impossible.

**Alternative considered.** A single `.claude/` directory with Cursor-specific generation only. Rejected because it leaks Claude Code's idioms into the source format and makes harness #3 awkward to add.

**Revisit if.** A future harness has primitives so fundamentally different that no universal source format works (e.g., a config-as-code harness with no markdown primitive).

### D2: Adapter per harness, not universal format with mechanical export

**Decision.** Each harness gets its own function in `scripts/sync-harnesses.js` — `syncClaude()`, `syncCursor()`. The function knows the target's file layout, frontmatter shape, and any required manifests.

**Why.** Harnesses differ in non-trivial ways: Cursor has no skill concept and uses `.mdc` rules; Claude Code wants frontmatter with `tools` and `model` fields. A "universal format then mechanical mapping" approach would need a least-common-denominator format that captures none of these well.

**Alternative considered.** A YAML manifest describing each agent/skill, with per-harness templates. Rejected as more machinery than the problem warrants for two harnesses.

**Revisit if.** Harness count exceeds four and the adapters start duplicating logic that could live in a shared layer.

### D3: Claude Code-style frontmatter as the source format

**Decision.** Source files use the frontmatter shape Claude Code expects (`name`, `description`, `tools`, `model` for agents; `name`, `description` for skills). Other harnesses translate from this.

**Why.** Claude Code's format is the most expressive of the harnesses agentry supports — it carries fields (`tools`, `model`) that other harnesses do not have. Translating *down* from a richer format loses information cleanly. Translating *up* from a lossy format guesses.

**Alternative considered.** Cursor's `.mdc` format as the source. Rejected because Cursor's frontmatter is closer to "rules" with less structure, and adding the Claude Code fields back would feel grafted on.

**Revisit if.** Claude Code's frontmatter schema changes substantially, or another harness emerges with a strictly richer format.

### D4: Cursor skills become `.mdc` rules with `alwaysApply: false`

**Decision.** Source skills (`skills/<name>/SKILL.md`) translate to Cursor rules at `.cursor/rules/<name>.mdc` with `alwaysApply: false` added to the frontmatter.

**Why.** Cursor has no first-class skill concept. The closest primitive is a project rule. Setting `alwaysApply: false` makes the rule available on-demand rather than injected into every context, which matches how skills are meant to be used.

**Alternative considered.** Skip Cursor for skills entirely. Rejected — that strands a real part of the value prop.

**Trade-off.** The Cursor rendering of a skill has lower fidelity than the Claude Code rendering. This is openly acknowledged in the `toCursorRule` comment in `scripts/sync-harnesses.js`.

**Revisit if.** Cursor ships a skill primitive, or `alwaysApply: false` stops behaving as expected.

### D5: Only generated subdirs of `.claude/` are wiped, not the parent

**Decision.** `syncClaude` wipes `.claude/agents/`, `.claude/skills/`, `.claude/commands/`. It does *not* wipe the parent `.claude/` directory.

**Why.** Claude Code stores per-user state — notably `settings.local.json`, the permissions cache — at the top level of `.claude/`. Wiping the parent would destroy that state on every sync.

**Alternative considered.** Wipe the whole `.claude/` and let Claude Code regenerate state. Rejected — silently destroying a user's permission grants is a hostile default, and the regeneration is not automatic.

**Discipline rule for future adapters.** Wipe what you own. Leave alone what you do not. Documented in `architecture.md` under "The settings preservation pattern."

### D6: `agentry-` prefix on Cursor agent filenames

**Decision.** Agents synced to `.cursor/agents/` are renamed to `agentry-<original-name>.md`.

**Why.** Cursor's `.cursor/agents/` directory is shared with the user's own project agents. Without the prefix, an agentry-shipped `planner` collides with a user's hand-written `planner`.

**Alternative considered.** Sync to a subdirectory like `.cursor/agents/agentry/`. Rejected because Cursor's agent discovery does not recurse into subdirectories by default.

## Content model

### D7: Companion agent + skill pairs

**Decision.** Some competencies ship as both an agent and a skill — `debugger` agent + `error-debugging` skill, `code-reviewer` agent + `code-review` skill.

**Why.** The two forms serve different contexts. An agent is for delegating to a fresh-context Claude instance for focused investigation. A skill is for shaping the current conversation's discipline when the same topic comes up mid-flow. Forcing one to substitute for the other compromises both.

**Trade-off.** Two files to maintain per competency. Mitigated by keeping the skill version shorter and reusing the same vocabulary the agent uses.

**Revisit if.** Usage data shows one form is consistently unused. At that point retire the unused half.

### D8: Commands are Claude Code-only

**Decision.** Slash commands (`/plan`, `/review`, etc.) sync to `.claude/commands/` only. Cursor does not receive them.

**Why.** Cursor has no slash-command system. Inventing a Cursor-side approximation (e.g., a "command" that is actually a snippet) would introduce confusing per-harness UX asymmetry without solving the underlying gap.

**Alternative considered.** Generate prompt-template files for Cursor. Rejected — the entire value of a command is the harness's invocation UX, which a template does not provide.

**User-visible trade-off.** Cursor users invoke the underlying agent directly. The README's "What's inside" section calls this out so the asymmetry is not a surprise.

**Updated: v0.7.** Commands now also sync to **OpenCode**, the other harness with a user-extensible command primitive. The "Claude Code-only" decision still holds for Cursor and Codex (no command primitive), but no longer for OpenCode — and the trade-off above applies to Cursor and Codex users only.

### D9: Universal content only in v0.1 and v0.2

**Decision.** Agents and skills are language- and framework-neutral. No TypeScript-specific examples, no React-specific guidance.

**Why.** The audience for v0.1 and v0.2 is "everyone." Universal content is useful to everyone; language-specific content is useful only to that language's users and is dead weight for everyone else.

**Trade-off.** Universal content is less concrete. The compensating tactic is to be vivid about *patterns* (off-by-one errors, missing null checks) rather than syntax.

**Revisit when.** v0.3 introduces language packs — separate opt-in directories per language. Authored separately, synced separately, installable separately.

**Updated: v0.11.** No language packs have shipped. All content through v0.11 remains language- and framework-neutral, and the curated-neutral positioning is reaffirmed by D17. Language packs remain a possible future direction, not a current one.

### D10: Curated over comprehensive — quality over quantity

**Decision.** Every agent and skill must earn its place. Five excellent skills beats fifty mediocre ones. "Might be useful" is not enough to ship.

**Why.** A sprawling library users stop trusting after a month is worse than a tight one they read end-to-end. This is the positioning that separates agentry from maximalist alternatives.

**How it shows up in practice.** New content needs a one-line statement of the concrete problem it solves. CLAUDE.md and `docs/authoring.md` both repeat this constraint.

**Revisit if.** The positioning shifts — for instance, if agentry pivots to being a comprehensive marketplace. The repo would need to reorganize, not just expand.

## Tooling

### D11: Zero runtime dependencies

**Decision.** `scripts/sync-harnesses.js`, `scripts/doctor.js`, and `scripts/lint-frontmatter.js` use Node stdlib only. No `js-yaml`, no `chalk`, no `commander`.

**Why.** A configuration framework with a sprawling dependency tree inverts its own value proposition. Stays auditable. `git clone && node scripts/sync-harnesses.js` works without `npm install`.

**Revisit if.** A genuine need emerges that stdlib cannot satisfy cleanly. The default answer is "write it inline."

### D12: Shell installers, not a Node CLI

**Decision.** Install is `scripts/install.sh` (POSIX) and `scripts/install.ps1` (PowerShell). No `bin/agentry` Node entry point.

**Why.** The install operation is "copy these directories to a known location." A Node CLI would need argument parsing, help text, error formatting — all overhead for a one-line operation. Shell scripts wrap it transparently and are auditable in 30 seconds.

**Revisit if.** Install grows to need cross-platform logic shell cannot express cleanly, or the harness count makes one unified CLI more pleasant than parallel installers.

### D13: `doctor` and `lint` as separate scripts

**Decision.** `npm run doctor` checks "is my install healthy?" — sources present, generated matches source, user-install reachable. `npm run lint` checks "is my source valid?" — frontmatter shape and required fields.

**Why.** The two audiences are different. `doctor` is for users debugging a broken install. `lint` is for authors checking their work before commit. Bundling them would force one audience through checks they do not need.

### D14: CI checks sync drift, not functionality

**Decision.** `.github/workflows/sync-check.yml` runs `npm run sync && git diff --exit-code` and `npm run lint`. It does not run unit tests.

**Why.** v0.2 has no unit tests to run. The most common contribution mistake is forgetting to commit regenerated files; the second most common is malformed frontmatter. CI catches both. Adding test infrastructure before there are tests to run would be premature.

**Revisit when.** Unit tests land for `toCursorRule` and the frontmatter parser (planned for v0.3). At that point CI grows an `npm test` step.

**Updated: v0.3.** Unit tests for `toCursorRule` and the frontmatter parser landed in v0.3 Bundle 1. The CI workflow now runs `npm test` as a third job alongside `sync-determinism` and `frontmatter-lint`.

## Process

### D15: Conventional commits enforced by convention, not tooling

**Decision.** Commit messages follow `feat:`, `fix:`, `refactor:`, `docs:`, `chore:`. Enforcement is human review and CLAUDE.md guidance — no `commitlint`, no Husky hook.

**Why.** A small project does not warrant the infrastructure. Convention scales until the contributor count grows.

**Revisit when.** Contributor count crosses ~5 and human review of every commit message becomes a bottleneck.

### D16: No v1.0 yet — staying pre-1.0 deliberately

**Decision.** v0.1 shipped. v0.2 shipped. v1.0 is not yet on the roadmap.

**Why.** v1.0 is a commitment to a stable adapter contract. agentry has shipped two adapters; the shape that supports four or five (Codex, OpenCode, Zed, Antigravity) is not yet known. Committing to "no breaking changes" before the shape stabilizes would be premature.

**Revisit when.** Five or more harness adapters are in the tree and have not required schema changes for two consecutive minor releases.

**Updated: v0.11.** Four adapters have now shipped — Claude Code, Cursor, Codex, and OpenCode — without the source schema changing. The revisit trigger (five adapters, stable across two consecutive minors) is closer but not yet met; agentry stays deliberately pre-1.0.

## Strategy and scope

### D17: Curated alternative to ECC, not an ECC competitor

**Decision.** agentry positions as the smaller, sharper, easier-to-read alternative for developers who use multiple AI tools and want a configuration set they can actually read end-to-end. Not as a maximalist all-in-one trying to cover every use case.

**Why.** ECC (the dominant maximalist project) already does maximalism well — hundreds of skills, dozens of agents, a Pro-tier funding model, an active contributor base. A solo-built clone of that positioning would lose on volume and lose on momentum. The alternative niche — a tight, opinionated, multi-harness convenience layer that does fewer things well — is structurally available because maximalist projects cannot serve it without abandoning their own positioning.

**Alternative considered.** Build agentry as a maximalist competitor, betting that better curation or cleaner UX could overtake ECC over time. Rejected because (a) the time investment to reach parity is multi-year, (b) the differentiation would be hard to communicate ("ours is better-built" is weaker than "ours is for a different audience"), and (c) it inverts the project's actual strength.

**Trade-off.** The narrower positioning means agentry will never serve users who want a comprehensive one-stop. That is a deliberate boundary, not a gap.

**Revisit if.** ECC pivots to a position that vacates the curated niche, or agentry's contributor base grows enough that maximalist breadth becomes maintainable.

### D18: v0.1 shipped with one agent and one skill — the minimum to prove the pattern

**Decision.** v0.1 shipped with exactly one agent (`code-reviewer`) and one skill (`tdd-workflow`), plus the sync engine and installers. Resisted the temptation to ship 5-10 of each to look more substantive.

**Why.** The point of v0.1 was to prove the source-of-truth + sync + install pattern works end-to-end across two harnesses. Shipping more content would have added review burden without changing whether the pattern is sound. Two well-written examples set the quality bar; ten mediocre ones lower it.

**Alternative considered.** Ship v0.1 with 5+ agents and 5+ skills to feel more "real" at launch. Rejected — the discipline of shipping less is what differentiates agentry from maximalist alternatives. Starting with restraint signals what the project values.

**Revisit if.** The minimum-viable approach turns out to leave first-time users with nothing useful to do. So far, evidence suggests two well-built examples are enough to demonstrate the framework's value.

### D19: v0.2 shipped Tier 1 + Tier 2 in parallel, without dogfooding v0.1 first

**Decision.** v0.2 expanded to four agents, five skills, five commands, plus CI workflow, doctor/lint scripts, and contributor/authoring docs — all in one release, without using v0.1 in real workflows first to identify gaps.

**Why.** Maintainer judgment that forward motion was more valuable than evidence-driven prioritization at this stage. The expansion stayed defensible by holding two constraints: every component remained universal (no language-specific content) and every component had a one-line justification before shipping.

**Alternative considered.** Dogfood v0.1 for 2-4 weeks, capture findings, and use them to prioritize v0.2 content. Recommended by the design-review process; not chosen. The risk: some v0.2 components may be solving problems the maintainer doesn't actually have, and won't get caught until later usage surfaces the gap.

**Mitigation.** Universal content is harder to be wrong about than stack-specific content. The v0.2 components are all generic enough to be useful in a wide range of workflows even if the maintainer hadn't personally needed them yet.

**Revisit if.** Real usage shows specific v0.2 components are never invoked. At that point retire them rather than letting them inflate the library.

## MCP

### D20: MCP servers as a content type — one neutral source, per-harness shape, root config files treated as owned

**Decision.** MCP server configs are a first-class source type at `mcp/<name>.json`. Each file is the server *definition* as it appears inside Claude/Cursor's `mcpServers` map; the filename is the server name. Sync emits the harness-native config for the three harnesses that take a JSON map: Claude Code (`.mcp.json`, repo root), Cursor (`.cursor/mcp.json`), and OpenCode (`opencode.json`, repo root, under the `mcp` key). Codex is skipped.

**Why a content type at all.** An MCP server is declared in a different file for every harness (`.mcp.json`, `.cursor/mcp.json`, `opencode.json`) but describes the same process or endpoint. That is precisely the maintained-twice-and-drifts problem agentry exists to kill — and unlike agents or skills, the content is structured config, not prose, so the translation is loss-free.

**Why the filename is the name.** MCP definitions carry no name field of their own — the name is the map key. Deriving it from the filename matches how agents and skills take their identity (D3) and keeps the source as the bare definition.

**Why a merged output, not one file per source.** This is the first content type whose generated form is a single file aggregating many sources, because that is the shape the harnesses read. The merge is sorted by name so the output is byte-stable regardless of `readdir` order — the same idempotence (D1, CI clean-tree) every other type gets for free.

**Why Claude and Cursor pass through but OpenCode transforms.** Claude Code and Cursor both read a `mcpServers` map whose entries are exactly the neutral definition, so the adapter wraps and writes verbatim. OpenCode's schema differs — `type: "local" | "remote"`, a single `command` array (command + args), an `environment` map (not `env`), an `enabled` flag — so OpenCode needs a real transform (`toOpenCodeMcpConfig`), the same way Cursor needs one for skills (D4).

**Why the root config files are treated as owned, not merged into.** `.mcp.json` and `opencode.json` are the only generated artifacts that land outside a harness namespace directory (`.claude/`, `.cursor/`, `.codex/`, `.opencode/`) — they are the project-scope paths the harnesses read from the repo root. agentry treats them as fully generated: you add servers as `mcp/*.json` sources, never by editing the outputs. But unlike the namespace directories, agentry does **not** wipe them — each is written only when sources exist and is never deleted, so a config a user wrote by hand (notably `opencode.json`, which holds far more than MCP) is not clobbered by an empty `mcp/`. "Wipe what you own" (D5) becomes "write what you own, but don't delete what you might not have created."

**Why Codex is deferred.** Claude, Cursor, and OpenCode all take a JSON map. Codex stores servers as TOML under `[mcp_servers.<name>]` inside its shared `config.toml` — the same file holding model, approval, and sandbox settings. Emitting that safely needs a TOML serializer and a merge that preserves the user's other keys, its own design problem. Deferred, in the same spirit as Codex rules historically were.

**Alternative considered — make each source a complete `.mcp.json` fragment.** Rejected: it repeats the `mcpServers` wrapper in every file and reintroduces a name that must track the filename. The bare-definition form is smaller and has a single source of the name.

**Install is deferred.** Sync produces the native files; the installers (D12) are not extended to place them. Installing MCP means *merging* into a user's existing config (`~/.claude.json`, a project `.mcp.json`, `.cursor/mcp.json`, `opencode.json`) without destroying servers they added themselves — a stateful merge, not the directory-copy the installers do today. Shipping a copy that silently overwrote a user's config would be the hostile default D5 warns against. Deferred until the merge is designed.

**Revisit if.** Codex MCP support lands (needs the TOML path), or install grows a safe-merge step that can place MCP files without clobbering user servers.

## Release

### D21: Release cuts a GitHub Release from the tag and CHANGELOG — no npm publish

**Decision.** `.github/workflows/release.yml` triggers on a `v*` tag push. It verifies the tag matches `package.json`'s version, re-runs the full gate (sync-drift check, `npm run lint`, `npm test`), extracts the matching `## [X.Y.Z]` section from `CHANGELOG.md`, and creates a GitHub Release with that section as the notes. It does **not** publish to npm.

**Why no npm publish.** agentry installs by `git clone` + an install script (D12), not by `npm install agentry`. There is no package on the registry to publish. A publish step would imply a distribution channel the project does not use, and would need registry credentials for no benefit. The GitHub Release *is* the distribution artifact: a tagged, documented point users can clone.

**Why regenerate the notes from CHANGELOG instead of writing them in the workflow.** The CHANGELOG is already the human-authored, curated record of each release (D10 discipline applied to release notes). Extracting the matching section keeps a single source of truth — the notes on GitHub are exactly what the CHANGELOG says, and cannot drift from it.

**Why gate the tag on sync/lint/test.** A tag is a stronger claim than a push — it says "this is a release." Re-running the same checks CI runs on every push (D14) at the tagged commit guarantees a release can never be cut from a tree that drifts, fails lint, or fails tests, even if the tag is pushed from a stale local checkout. The tag-version/package-version match is the guard against tagging the wrong commit.

**Why only stdlib and `gh`.** Consistent with D11's zero-dependency spirit: the workflow uses `actions/checkout`, `actions/setup-node`, and the `gh` CLI preinstalled on the runner — no third-party marketplace actions whose supply chain agentry would be trusting to cut its releases.

**Revisit if.** agentry starts distributing through a package registry (then a publish job joins the release), or the release process grows steps (signing, provenance) that warrant a dedicated reusable workflow.
